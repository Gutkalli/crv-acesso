1. VISÃO GERAL DA ARQUITETURA
================================================================================

O backend segue o padrão de arquitetura em camadas (Layered Architecture),
onde cada camada tem uma responsabilidade única e se comunica apenas com a
camada adjacente:

  Requisição HTTP
       ↓
  [routes/]           — Define endpoints e aplica middlewares
       ↓
  [middlewares/]       — Valida JWT, verifica perfil, loga latência
       ↓
  [controllers/]       — Valida input, orquestra a resposta
       ↓
  [services/]          — Lógica de negócio e acesso ao Supabase
       ↓
  [SupabaseService]    — Cliente HTTP para a API REST do Supabase
       ↓
  Supabase (PostgreSQL + Auth)

Tecnologias usadas:
  - Flask 3.0.3              — Framework web Python
  - Flask-CORS 4.0.0         — Cross-Origin Resource Sharing
  - Flask-Limiter 3.5.0      — Rate limiting por IP
  - python-dotenv 1.0.1      — Carregamento de variáveis .env
  - urllib (stdlib)           — Cliente HTTP puro, sem dependências extras
  - Supabase REST API         — Banco de dados PostgreSQL + Auth JWT

3. CAMADA: config.py
================================================================================

Carrega as variáveis de ambiente do arquivo .env e expõe como atributos de
classe, disponíveis em todo o projeto via "from config import Config".

Variáveis obrigatórias no .env:
  SUPABASE_URL              URL do projeto Supabase (ex: https://xxx.supabase.co)
  SUPABASE_ANON_KEY         Chave anon/public do Supabase
  SUPABASE_SERVICE_ROLE_KEY Chave service_role (admin total, sem RLS)

Variáveis opcionais (com defaults):
  SECRET_KEY                Chave secreta Flask (default: crv-secret-dev-2025)
  FLASK_DEBUG               true/false (default: false)
  PORT                      Porta do servidor (default: 10000)
  CORS_ORIGINS              IPs/domínios permitidos (vírgula separados)
  RATE_LIMIT_DEFAULT        200 per minute
  RATE_LIMIT_AUTH           10 per minute
  RATE_LIMIT_SENSITIVE      5 per minute
  RATE_LIMIT_EXPORT         20 per minute

4. CAMADA: views/

RESPONSE.PY
-----------
Define o formato padrão de todas as respostas JSON da API:

  { "success": true/false, "data"/"error": ..., "message"?: ...,
    "pagination"?: ..., "timestamp": "ISO8601" }

Funções disponíveis:
  success(data, message, status, pagination) → 200
  created(data, message)                      → 201
  no_content()                                → 204
  paginated(data, total, page, per_page)      → 200 com metadados de paginação
  error(message, status, detail, code)        → qualquer 4xx/5xx
  bad_request(message, detail)                → 400
  unauthorized(message)                       → 401
  forbidden(message)                          → 403
  not_found(resource)                         → 404
  unprocessable(message, detail)              → 422
  server_error(message, detail)               → 500

ERROS.PY
--------
Registra handlers globais para Flask via register_error_handlers(app).
Cobre os códigos: 400, 401, 403, 404, 405, 422, 429, 500.
Garante que mesmo erros não tratados retornem JSON, nunca HTML.

5. CAMADA: services/
================================================================================

SUPABASE_SERVICE.PY
-------------------
Cliente HTTP de baixo nível para o Supabase REST API.
Usa urllib da stdlib — sem dependências externas.

Autenticação por header:
  - Sem token: usa SUPABASE_SERVICE_ROLE_KEY → bypassa RLS (operações admin)
  - Com token: usa SUPABASE_ANON_KEY + Bearer token do usuário → respeita RLS

Métodos HTTP:
  get(path, token)
  post(path, payload, token, prefer)
  patch(path, payload, token)
  delete(path, token)

Atalhos para tabelas REST:
  select(tabela, filtros, select, order, limit, offset, token)
  insert(tabela, payload, token)
  update(tabela, filtro, payload, token)
  remove(tabela, filtro, token)
  count(tabela, filtros, token)

BASE_SERVICE.PY
---------------
Classe pai de todos os services de domínio.
Instancia self.sb = SupabaseService() automaticamente.

AUTH_SERVICE.PY
---------------
Autentica usuários via Supabase Auth.
Métodos:
  login(email, senha)              → /auth/v1/token?grant_type=password
  logout(token)                    → /auth/v1/logout
  usuario_logado(token)            → /auth/v1/user
  buscar_perfil_db(uid, token)     → tabela usuarios (verifica ativo=true)
  recuperar_senha(email)           → /auth/v1/recover (sempre retorna sucesso)
  alterar_senha_logado(token, pwd) → PATCH /auth/v1/user
  enroll_2fa(token)                → /auth/v1/factors (TOTP)
  challenge_2fa(token, factor_id)  → /auth/v1/factors/{id}/challenge
  verify_2fa(token, fid, cid, cod) → /auth/v1/factors/{id}/verify + atualiza DB
  unenroll_2fa(token, fid, uid)    → DELETE /auth/v1/factors/{id} + atualiza DB

USUARIOS_SERVICE.PY
-------------------
CRUD completo de usuários do sistema (tabela `usuarios`).
Métodos:
  listar(token)
  buscar(uid, token)
  criar(dados, perfil_solicitante)  → cria no Auth primeiro, depois no DB.
                                      Se DB falhar: rollback no Auth (delete).
  atualizar(uid, dados, token)
  alterar_senha(uid, nova_senha)    → PATCH /auth/v1/admin/users/{uid}
  deletar(uid, solicitante_id)      → bloqueia auto-deleção

FUNCIONARIOS_SERVICE.PY
-----------------------
CRUD + importação em lote de funcionários (tabela `funcionarios`).
Campos aceitos: nome, cpf, matricula, cargo, status, setor, turno,
                email, telefone, empresa_id
Método importar(registros, token): lotes de 50 registros para evitar timeout.

EQUIPAMENTOS_SERVICE.PY
-----------------------
CRUD de equipamentos (tabela `equipamentos`).
Método testar_conexao(ip, porta, usuario, senha):
  Faz POST HTTP para http://{ip}:{porta}/systemManager.fcgi?session=0
  Retorna {"online": bool, "codigo_http": int} ou {"online": false, "erro": str}
  Timeout: 5 segundos.

CREDENCIAIS_SERVICE.PY
----------------------
CRUD de credenciais de acesso (tabela `credenciais`).
Tipos: cartão, biometria, qrcode, pin
Campos: funcionario_id, tipo, identificador, status, validade, empresa_id

OCORRENCIAS_SERVICE.PY
----------------------
CRUD de incidentes de segurança (tabela `ocorrencias`).
Campos: tipo, prioridade (alta/media/baixa), status (aberta/em_andamento/fechada),
        descricao, local, funcionario_id, responsavel_id, data_ocorrencia

ACESSOS_SERVICE.PY
------------------
Log de passagens (tabela `acessos`). Maioritariamente leitura.
Método stats_hoje(token, data_ref):
  Retorna {total, negados, liberados} para uma data específica.
  Usa count() em vez de buscar registros completos — mais eficiente.

AUDITORIA_SERVICE.PY
--------------------
Log imutável de ações sensíveis (tabela `auditoria`).
SEMPRE usa service_role (sem token) para garantir que o log não passe por RLS.
Níveis: info, aviso, critico
Campos: usuario_id, usuario_email, acao, modulo, descricao, nivel, data, dados_extras

REGRAS_ACESSO_SERVICE.PY
------------------------
CRUD de regras de controle de acesso (tabela `regras_acesso`).
Campos: nome, descricao, equipamento_id, setor, cargo, turno,
        horario_inicio, horario_fim, dias_semana, ativo, empresa_id

CONFIGURACOES_SERVICE.PY
------------------------
Upsert de configurações globais (tabela `configuracoes`).
Método salvar(dados, token, empresa_id):
  Se já existe registro para empresa → PATCH.
  Se não existe → INSERT.

6. CAMADA: auth/

SUPABASE_AUTH.PY
----------------
Funções puras para validação de token e busca de perfil.
Usadas pelo middleware de autenticação.

extrair_token(authorization_header)
  → Extrai o Bearer token do header.
  → Retorna (token, erro).

validar_token(token)
  → GET /auth/v1/user com o token.
  → Retorna (auth_data, erro). auth_data contém id, email, role.

buscar_perfil(user_id, token)
  → SELECT na tabela usuarios para pegar perfil, ativo, empresa_id.
  → Verifica se o usuário está ativo.
  → Retorna (usuario_dict, erro).

7. CAMADA: middlewares/
================================================================================

AUTH_MIDDLEWARE.PY
------------------
Decoradores de autenticação para uso nas rotas.

_autenticar()
  Função interna. Chama extrair_token → validar_token → buscar_perfil.
  Popula g.token, g.user_id, g.user_email, g.user_perfil, g.empresa_id.
  Retorna (True, None) ou (False, response_401).

@requer_auth
  Exige apenas que o JWT seja válido. Qualquer perfil pode passar.
  Popula g.* com dados do usuário.

@requer_perfil("admin", "gerente")
  Exige autenticação + pelo menos um dos perfis listados.

@requer_admin
  Atalho para requer_perfil("admin", "gerente").

@somente_admin
  Atalho para requer_perfil("admin").

PERMISSAO.PY
------------
Funções puras de regras de negócio de autorização.
Retornam sempre (bool, str_motivo).

pode_criar_usuario(perfil_solicitante, perfil_alvo)
  Admin cria qualquer perfil.
  Gerente cria apenas operador e portaria.
  Outros perfis não podem criar usuários.

pode_editar_usuario(perfil_solicitante, perfil_alvo)
  Mesma regra da criação.

pode_excluir_usuario(perfil_solicitante, uid_solicitante, uid_alvo)
  Somente admin pode excluir.
  Ninguém pode excluir a si mesmo.

pode_ver_auditoria(perfil)
  Apenas admin e gerente.

pode_alterar_configuracoes(perfil)
  Apenas admin.

pode_gerenciar_regras(perfil)
  Admin e gerente.

LOGGER.PY
---------
init_logger(app)
  Configura logging.basicConfig com nível INFO (ou DEBUG se app.debug).
  Silencia logs barulhentos de urllib3 e werkzeug.

register_request_hooks(app)
  @app.before_request: salva g._start = time.monotonic()
  @app.after_request:
    - Loga "METHOD /path → STATUS [Xms] uid=... ip=..."
    - Adiciona headers de segurança: X-Content-Type-Options, X-Frame-Options,
      Referrer-Policy
    - Adiciona Cache-Control: no-store para /static/

BASE_CONTROLLER.PY
------------------
Utilitários usados por todos os controllers.

parse_json()             → request.get_json(force=True, silent=True) or {}
parse_pagination()       → (limit, offset) da query string (max 500)
parse_filtros_data(col)  → string de filtro "campo=gte.X&campo=lte.Y"
sanitize(valor, maxlen)  → strip + truncate

Cada controller segue o padrão:
  1. Lê body/params via parse_json() ou request.args
  2. Valida campos obrigatórios → retorna bad_request() se inválido
  3. Verifica permissão via middlewares.permissao se necessário
  4. Chama o service correspondente
  5. Mapeia erro "not_found" → 404, outros erros → 422
  6. Retorna success() / created() / no_content()

================================================================================
10. ENTRY POINT: app.py
================================================================================

Usa o padrão Application Factory (create_app()).
Vantagens: facilita testes, múltiplas instâncias, configuração dinâmica.

Ordem de inicialização dentro de create_app():
  1. Flask(__name__, static_folder="static")
  2. app.secret_key e app.debug do Config
  3. init_logger(app)               ← configura logging
  4. register_request_hooks(app)    ← before/after request
  5. CORS(app, ...)                 ← restringe origens
  6. Limiter(app, ...)              ← rate limiting global (200/min)
  7. register_error_handlers(app)   ← handlers JSON globais
  8. register_routes(app)           ← todos os blueprints

O objeto `app` é criado no módulo para compatibilidade com gunicorn:
  gunicorn app:app

13. FLUXO DE UMA REQUISIÇÃO (passo a passo)
================================================================================

Exemplo: GET /api/funcionarios?status=ativo&limit=20

1. Flask recebe a requisição na porta 10000

2. register_request_hooks → before_request:
   - Salva g._start = time.monotonic()

3. Flask despacha para o Blueprint "funcionarios" (url_prefix="/api/funcionarios")

4. Decorador @requer_auth na função listar():
   - Extrai "Bearer <token>" do header Authorization
   - GET /auth/v1/user com o token → Supabase retorna {id, email, ...}
   - SELECT na tabela `usuarios` → retorna {perfil, ativo, empresa_id}
   - Popula g.token, g.user_id, g.user_email, g.user_perfil, g.empresa_id

5. Chama funcionarios_controller.listar():
   - parse_pagination() → limit=20, offset=0
   - _montar_filtros() → "status=eq.ativo"
   - _svc.listar(token=g.token, filtros="status=eq.ativo")

6. FuncionariosService.listar():
   - self.sb.select("funcionarios", filtros="status=eq.ativo", order="nome.asc", token=token)

7. SupabaseService.select():
   - Monta path: /rest/v1/funcionarios?select=*&status=eq.ativo&order=nome.asc
   - _headers(token=user_token) → usa anon key + Bearer token (respeita RLS)
   - urllib.request.urlopen() → GET para Supabase
   - Retorna (list[dict], None)

8. Retorno sobe pelas camadas:
   - Service retorna (data, None) ao controller
   - Controller fatia data[0:20] para paginação, retorna success(pagina, pagination={...})
   - success() retorna jsonify({success:true, data:[...], pagination:{...}, timestamp:...}), 200

9. register_request_hooks → after_request:
   - Loga "GET /api/funcionarios → 200 [42ms] uid=xxx ip=127.0.0.1"
   - Adiciona headers de segurança à resposta

10. Flask envia resposta HTTP 200 ao cliente

14. SEGURANÇA
================================================================================

AUTENTICAÇÃO
  - JWT validado em cada requisição via Supabase /auth/v1/user
  - Token nunca armazenado no servidor (stateless)
  - Suporte a 2FA TOTP (Google Authenticator, Authy, etc.)

AUTORIZAÇÃO
  - Row Level Security (RLS) no Supabase ativo para operações com user token
  - service_role usado apenas para operações admin críticas (criar user, auditoria)
  - Hierarquia de perfis: admin > gerente > operador > portaria
  - Regras em middlewares/permissao.py — funções puras, testáveis

RATE LIMITING
  - 200 requisições/minuto por IP (padrão global)
  - Flask-Limiter com storage em memória (dev) ou Redis (prod)

CORS
  - Apenas origens explícitas via Config.CORS_ORIGINS
  - Nunca wildcard (*)

HEADERS DE SEGURANÇA (após cada resposta)
  - X-Content-Type-Options: nosniff
  - X-Frame-Options: DENY
  - Referrer-Policy: strict-origin-when-cross-origin

ROLLBACK DE AUTENTICAÇÃO
  - Se criação de usuário falha no DB após criar no Auth:
    DELETE /auth/v1/admin/users/{id} é chamado automaticamente

PROTEÇÃO DE ROTAS
  - Rotas GET de leitura: @requer_auth (qualquer usuário autenticado)
  - Rotas de escrita (POST/PUT/DELETE): @requer_admin (admin/gerente)
  - Rotas críticas (excluir usuário, config): @somente_admin
